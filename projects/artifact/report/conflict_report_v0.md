# 需求衝突報告

## CR-1：平台多方利益平衡與費用/分潤規則衝突

### 涉及需求
- URL-15：平台營運管理者需要系統的功能與費用設計能在吸引消費者、照顧外送員權益和維持平台收入三者之間取得平衡。
- URL-19：餐廳商家需要平台收取的手續費維持在合理範圍，以確保商家在薄利多銷情境下仍能獲利並維持續約及持續上架意願。
- URL-3：消費者需要外送平台能夠透明公開所有商品價格及相關費用（如運費、服務費），避免於結帳流程中出現未事先告知的額外費用，以提升價格透明度與消費信任。

### 衝突描述
平台多方利益平衡、商家手續費合理及消費者價格公開透明三項需求，均需共用同一費用設計與分潤決策規則，若無合併、優先順序或明確裁定，將於營運條件、收費透明度及利益分配產生邏輯矛盾，無法原樣共同納入規格書，必須裁定或調整。

### 解決選項
1. Stakeholder Negotiation：召開平台營運管理者、餐廳商家與消費者代表的協調會，明確釐清各方需求底線，討論並共識可接受的費用結構方案（如設定手續費上限、利潤分配比率與透明揭露機制），將最重要的優先原則寫入規格書。
2. Conditional Logic：根據不同用戶角色或情境定義差異化的費用規則（如新商家、有特定優惠的消費者、不同地區收費公式），將多方需求以條件方式納入系統設計，並針對每類費用出現透明公開的特殊例外條件。
3. Prioritization：依據平台整體策略，明確設定主優先需求（如優先保護商家存活、或優先保障消費者權益、或平台收入最大化），將次要需求適度調整至可配合主策略執行的範圍，逐步發佈不同優先級功能。

### 建議解法
建議優先採用 Stakeholder Negotiation，透過正式協議達成三方利益衡量下的費用與分潤方案，既能照顧各方關切又提升規範落地的可行性與穩定性；在協商無法取得跨方共識時，可輔以 Conditional Logic 處理特殊情境；如需推進速度，可再考慮 Prioritization，但此舉需審慎考量被壓縮方的長期合作意願與平台健康。

## CR-2：外送員派單公平與異常上報時之權益矛盾

### 涉及需求
- URL-10：外送員需要外送平台合理分配訂單，避免派單數量過少或集中特定外送員，以確保派案制度的公平性。
- URL-25：外送員需要能夠在遇到餐廳未營業或備餐延遲等異常情況時，於平台上快速上報並獲得明確的後續處理指引，以減少現場等待與不確定感，並確保派單量與收入不因異常事件受到不公平影響。

### 衝突描述
派單公平與異常上報後保障派單權益兩項需求，均針對同一派單演算法與收入分配提出限制，若未明確定義異常通報對派單的影響，將導致公平性與權益保障在流程與狀態轉移上產生邏輯衝突，無法直接共存，需合併或裁定。

### 解決選項
1. Conditional Logic：根據異常上報狀態，調整派單規則。例如：外送員如因異常（如餐廳未營業、備餐延遲）經確認後暫時不計入派單輪轉與考核，恢復正常時再納入公平分配。平台需明確定義各異常類型及生效範圍。
2. Decomposition：將派單公平性與異常情境的派單保障拆分為獨立模組。一般狀態下執行標準公平分配，遇到異常事件則由專門機制一律暫停派單、專案審核後單獨處理補償、追蹤異常事件對收入的影響。
3. Technical Solution：透過演算法優化，動態結合異常上報與派單公平權重。例如：遇異常時外送員派單權重暫時調整，後續同步異常審核狀態，自動補發遺失單量或收入補差。

### 建議解法
建議採用條件式邏輯（Conditional Logic）。此策略能在外送員因異常上報無法正常服務時，明確規範其派單狀態，兼顧公平分配原則與異常情境下的權益保護。可配合異常確認機制，提升流程透明度，且相對易於需求溝通。

## CR-3：外送員端即時提醒與不干擾操作的介面衝突

### 涉及需求
- URL-26：消費者需要外送平台在補充取餐地點或接收方式等指示後，能夠明確回饋外送員已讀取並確認相關訊息，以提升資訊傳遞的可靠性與消費者安心感。
- URL-28：外送員需要外送平台在接單後及接近目的地前，能夠明顯且即時提醒取餐或送餐的特殊指示，並確保相關內容不被隱藏，方便外送員在配送過程中即時掌握消費者的臨時留言或變更資訊。
- URL-31：外送員需要外送平台在送餐流程的每個關鍵步驟提供明顯且不易忽略的提醒，並確保重要訊息不會被隱藏於多層選單或頁面中，以降低資訊遺漏風險。
- URL-33：外送員需要外送平台的提醒功能不會妨礙導航、問題回報等其他重要操作，避免因提醒訊息導致操作誤觸或流程中斷。

### 衝突描述
外送員端提醒需即時、明顯且不可忽略，與提醒不得妨礙其他操作的需求，於通知設計與操作流暢性產生不可兼容的流程衝突，若無優先順序或例外條件，將導致界面行為矛盾，必須合併或調整細節後方可納入規格書。

### 解決選項
1. Conditional Logic：針對不同情境與任務流程設計『彈性提醒』：在關鍵步驟時彈出明顯提醒，但若外送員正進行導航或重要操作，則以較不干擾（如橫幅、振動、標記待讀等）提示，並於操作結束後再次推送完整訊息，確保訊息不漏接且不干擾操作。
2. Prioritization：針對提醒與操作流暢性孰輕孰重，由利害關係人協商並明確劃分優先順序：若強化訊息可見性為最高優先，允許提示打斷部分操作；如以外送員流暢完成操作為主，提醒訊息則以次要提示並允許手動確認。
3. Stakeholder Negotiation：召集外送員與消費者代表共同討論通知細節與優先規則，協作制定例外與原則，例如：僅對特定緊急變動以強提醒，其餘訊息採普通提醒，異常流程可人工上報協調，最終由平台做出調和設計。

### 建議解法
建議採用『情境條件式提醒（Conditional Logic）』，即動態調整提醒強度與方式，兼顧不漏接重要資訊與不妨礙操作安全與效率。此方案能靈活對應多元場景並維護雙方需求平衡。儘管開發難度較高，但更能提升用戶滿意度並減少未來潛在的衝突。

## CR-4：消費者彈性取餐地點與平台高風險場域責任揭露矛盾

### 涉及需求
- URL-21：消費者需要能夠在每次訂餐時自由選擇取餐地點與接收方式，並可於平台介面彈性補充取餐相關指示或注意事項，且外送員能確實收到這些資訊，以因應不同生活情境下的需求。
- URL-27：平台營運管理者需要外送平台在消費者指定非住家地點（如公司、超商、公共空間等）作為取餐地點時，能明確揭露相關風險與責任，並於必要時要求消費者在下單前確認同意條款、說明進出規則或提供聯絡協助，且若場域需要特別同意，平台必須能查驗消費者權限或留存同意證明，以避免平台承擔無法控管的糾紛責任。
- URL-30：平台營運管理者需要外送平台在涉及特殊或高風險取餐地點時，具備要求消費者每次下單時主動確認並說明進出權限及特殊指示、必要時上傳門禁資訊或臨時聯絡人，並留存消費者同意證據，以降低營運風險並能於主管機關或消費申訴時佐證平台已盡責。
- URL-36：平台營運管理者需要系統在消費者指定特殊取餐地點（如校園、醫院、商辦大樓等）時，能夠揭露該地點的進出限制或隱私規範，並要求消費者在下單前明確同意相關風險與現場規則，必要時提供臨時聯絡人或門禁資料，並保留同意紀錄，以確保平台不被推定為現場配送問題的唯一責任方。

### 衝突描述
消費者彈性指定取餐地點與高風險地點需責任揭露、同意及查驗等控管要求，於流程彈性與責任分界產生根本矛盾，若無法合併條件或依法令裁定優先順序，將導致執行流程衝突，無法原樣共存。

### 解決選項
1. Conditional Logic：針對非住家、特殊或高風險的取餐地點（如公司、公共空間、校園、醫院等），平台自動啟用額外揭露、同意及查驗流程；一般住家則維持彈性快速取餐介面。根據訂單內容動態啟用不同取餐流程。
2. Prioritization：明確優先保障平台營運管理者對特殊取餐地點的風險控管需求，所有非住家地點強制執行揭露、同意及查驗權限等嚴格措施，即使降低消費者彈性。
3. Stakeholder Negotiation：召集消費者與平台營運管理者進行協商，針對高風險地點具體清單進行共識定義，部分中低風險地點可由消費者自負責任簡化流程，由主管機關或合格第三方協助劃分責任與流程細節。

### 建議解法
建議採用 Option A（條件式邏輯）：能維持消費者彈性體驗，同時確保高風險場域有足夠控管措施；清楚區分普通地點與特殊地點的責任分界，利於系統落地與合規經營。

## CR-5：消費者補償與外送員爭議流程狀態衝突

### 涉及需求
- URL-29：消費者需要外送平台在外送員未依照已讀的取餐指示完成時，能主動通知消費者、提供外送員說明未能配合原因的機會，並協助消費者了解狀況與處理流程，若無合理理由未配合時，平台需有具體補償機制以保障消費者權益。
- URL-34：消費者在外送員未依照補充取餐指示導致問題時，需要能夠於平台上簡單快速地選擇退款或等值補償（如折價券、平台點數），且補償金額需合理反映實際損失，並可依個人偏好選擇補償方式。
- URL-37：外送員在無法依照消費者補充指示完成配送或現場遇到突發狀況時，需要外送平台提供簡單明確且可快速操作的回報與協助流程，包含常見問題選項、即時留言、消費者即時通知與回應機制，以及在消費者未回應時由平台自動或人工協助處理，並於整個過程中提供進度提醒與紀錄，確保外送員權益不受損。

### 衝突描述
消費者補償、補償方式選擇與外送員權益保障三項需求，涉及同一爭議判定、補償流程與責任分工，若無明確細則約束，將於計算時機、補償方式、流程進入條件與訊息責任產生狀態與權益分化互斥，需整併或裁定後方可納入規格書。

### 解決選項
1. Decomposition：將消費者補償、補償方式選擇、外送員權益三大需求拆解為獨立模組，分別處理（1）消費者的即時補償流程設計，（2）補償方式選擇及其金額與核定依據，（3）外送員申訴與回報權益保護。針對互斥部分（如對同一事件如何裁決），再設立獨立決策邏輯或仲裁流程。
2. Conditional Logic：設計條件式流程，釐清不同情境下（如外送員回報合理/不合理原因、消費者選擇不同補償方式、平台是否即時辨識問題）所觸發的補償、通知及權益保護路徑。例如：若外送員提供合理解釋平台自動排除補償，否則自動啟動消費者補償並允許外送員申訴。
3. Technical Solution：以技術手段整合所有權益判斷、通知、補償機制與即時狀態同步。導入自動決策系統或協作平臺，介接消費者、外送員、平台三方互動紀錄與判斷依據，建立完整 trace 與透明機制，發生異常時能追溯並調查。

### 建議解法
建議採用『條件式流程（Conditional Logic）』，將三方需求整合進單一自動化流程，根據外送員回報、消費者選擇、平台判斷與即時回饋設計彈性且明確的流程路徑。此方案兼顧權益公平、彈性與效率，並易於後續擴充與釐清責任歸屬。
